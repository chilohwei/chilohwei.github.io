## 太长不读

最近整理一套多 Agent 协作架构，越看越觉得一个判断很重要：

**多 Agent 的重点，不是让更多模型同时工作，而是设计一套能协作、能恢复、能验收、能审计的组织系统。**

如果只是开多个 Agent，让它们互相发消息，最后很容易变成：

- 上下文越传越变形。
- 角色边界越来越模糊。
- 临时任务变成长期职责。
- 记忆互相污染。
- 工具权限失控。
- 最终结果看起来完整，但没人说得清楚怎么来的。

我现在会把多 Agent 拆成六个核心对象：

| 对象 | 类比 | 主要职责 |
|---|---|---|
| `Channel` | 团队工作台 | 承载任务、上下文、事件、产物 |
| `Main Agent` | 协作治理者 | 路由、仲裁、风险、最终整合 |
| `Worker Agent` | 长期专业成员 | 在清晰边界内做专业交付 |
| `Task Subagent` | 临时协作资源 | 做单次、局部、可退出的小任务 |
| `Memory` | 组织知识库 | 保存项目事实、共享方法、角色经验 |
| `Tools / MCP` | 动作能力 | 在权限和审计约束下接触外部世界 |

![Agent Organization Map](https://chilohdata.s3.bitiful.net/blog/blog/agent-organization-map.svg)

这篇不是讲“怎么堆更多 Agent”，而是讲我现在会怎样理解一个可长期运行的 Agent 团队。

## 先说问题：多 Agent 很容易把混乱放大

很多人聊多 Agent，第一反应是分工。

一个做产品，一个做设计，一个写代码，一个做测试。

听起来很合理。

但真正跑起来会发现，只要没有组织结构，多个 Agent 并不天然更好。

单 Agent 至少只有一个上下文、一个责任主体、一个最终出口。

多 Agent 如果边界不清，问题会更麻烦：

| 现象 | 本质问题 |
|---|---|
| A 说任务是这样，B 理解成那样 | 任务空间不统一 |
| 每个 Agent 都在总结别人 | 信息层层转述 |
| Worker 开始替 Main 做决策 | 职责边界漂移 |
| 临时子任务继续拆临时子任务 | 治理失控 |
| 所有经验都写进长期记忆 | 记忆污染 |
| 输出很好看，但没人验收 | 完成定义太松 |

所以我现在不太把多 Agent 理解成“并发问题”。

它更像组织问题。

**不是开更多线程，而是搭一套协作秩序。**

## Channel：任务现场不要放在某个 Agent 脑子里

多 Agent 系统里，第一个关键对象是 `Channel`。

它不是某个 Agent，也不是长期记忆。

它更像当前任务的公开工作台。

Channel 里应该放：

- 用户目标和约束。
- 当前任务状态。
- 这次任务需要的上下文包。
- 任务分配、阻塞、风险、产物等事件。
- Worker 和 Subagent 返回的结构化结果。
- 最终交付前的验收状态。

为什么要有这个东西？

因为如果没有 Channel，所有信息都会通过某个 Agent 转述。

用户告诉 Main，Main 告诉 Worker，Worker 回 Main，Main 再总结给另一个 Worker。

这样一层层传下去，信息必然变形。

更麻烦的是，每个 Agent 都会在转述时加入自己的理解。最后大家好像在处理同一个任务，其实任务已经分叉了。

有了 Channel，任务本身变成一个共享对象。

Agent 不需要从另一个 Agent 的二手叙述里猜目标，而是直接看同一份任务现场。

这句话很关键：

**协作空间应该属于任务，而不是属于某个 Agent。**

## Main Agent：治理者，不是传话筒

有了 Channel 之后，`Main Agent` 的位置也要变。

它不应该是所有信息的强制中转站。

它更像协作治理者。

它负责：

| 责任 | 具体含义 |
|---|---|
| 路由 | 判断任务该交给谁 |
| 仲裁 | 多个 Worker 结论冲突时做取舍 |
| 风险 | 发现高风险动作、缺证据、缺决策 |
| 预算 | 控制并行、工具调用、临时子任务数量 |
| 验收 | 确认最终产物是否满足条件 |
| 记忆治理 | 判断什么可以沉淀，沉淀到哪里 |

这和“什么都亲自做”是两回事。

Main Agent 如果亲自替每个 Worker 做专业交付，很快又回到单 Agent 瓶颈。

好的 Main Agent 应该像一个负责人：

- 知道什么时候分派。
- 知道什么时候追问。
- 知道什么时候暂停等用户决策。
- 知道什么时候做最终整合。
- 也知道什么时候不要插手专业判断。

它的价值不是输出更多内容。

它的价值是让系统不乱。

## Worker Agent：长期专业成员，要有边界

`Worker Agent` 不是一次性 prompt。

它应该像团队里的长期成员。

长期成员有几个特征：

| 特征 | 对 Agent 意味着什么 |
|---|---|
| 稳定职责 | 知道自己负责什么、不负责什么 |
| 判断框架 | 用固定方法理解问题，而不是每次临场发挥 |
| 输入要求 | 缺少必要信息时会明确返回 |
| 输出契约 | 下游知道怎么使用它的结果 |
| 权限边界 | 能用什么工具，不能碰什么东西 |
| 经验沉淀 | 只把角色级经验写入自己的记忆 |

比如产品型 Worker 负责问题定义、目标、范围、用户故事、验收标准。

设计型 Worker 负责信息结构、视觉层级、组件状态、交互体验。

工程型 Worker 负责实现路径、技术约束、可维护性、质量风险。

测试型 Worker 负责边界场景、验收用例、缺陷风险、发布建议。

这不是为了模仿公司组织架构。

而是为了让责任可以被定位。

如果一个 Agent 什么都能做，它就很难知道什么时候不该做。

真正成熟的 Worker，不是会说“我是某某专家”。

而是能做到：

- 任务不属于自己时，拒绝或升级。
- 输入不够时，明确说缺什么。
- 输出时带假设、风险和下一步交接。
- 需要临时帮助时，只拆局部子任务。
- 长期记忆只写和自己角色有关的经验。

**边界不是限制能力，边界是建立信任。**

## Task Subagent：临时帮手，不要变成新部门

`Task Subagent` 适合做很小、很明确、可以退出的任务。

比如：

- 查一组资料。
- 读几份文件。
- 跑一次测试。
- 压缩一段长上下文。
- 检查一份交付物有没有遗漏。
- 把结果整理成表格。

它和 Worker 最大的区别是生命周期。

Worker 是长期角色。

Subagent 是临时资源。

我会用这张表判断：

| 问题 | 如果答案是“是” |
|---|---|
| 这个任务是否只服务当前目标 | 可以考虑 Subagent |
| 是否有清晰输入和输出 | 可以考虑 Subagent |
| 是否可以独立完成，不影响主线 | 可以考虑 Subagent |
| 是否需要长期积累专业经验 | 应该是 Worker |
| 是否会接管最终判断 | 不应该是 Subagent |
| 是否需要写长期记忆 | 不应该是 Subagent |

Subagent 最怕“无限套娃”。

一个临时 Agent 查资料，查着查着又叫另一个分析，分析 Agent 又叫第三个总结。最后没人知道原始目标是什么，也没人对最终结果负责。

所以 Subagent 的规则应该很硬：

**只拿必要上下文，只做一个小任务，只返回结果，不接管主任务，不写长期记忆。**

## 记忆：不是越多越好，而是越有边界越好

Agent 一旦长期运行，记忆一定会变成核心问题。

但记忆不是垃圾桶。

什么都记，最后等于什么都不可信。

我现在会把记忆分成几类：

| 记忆类型 | 应该保存什么 | 不应该保存什么 |
|---|---|---|
| 用户偏好 | 多次出现或明确确认的稳定偏好 | 某次随口表达 |
| 项目记忆 | 目标、范围、决策、状态、产物 | 角色个人经验 |
| 共享知识 | 可复用方法、模板、最佳实践 | 未审核的临时猜测 |
| 原始资料 | 外部材料、访谈、抓取结果 | 系统指令 |
| 角色私有记忆 | 某个 Worker 的角色级经验 | 其他 Worker 的私有笔记 |

这里最重要的是隔离。

产品 Worker 的经验，不应该直接变成工程 Worker 的规则。

测试 Worker 在某个项目里发现的风险，也不等于所有项目都要照搬。

跨角色共享当然需要，但要通过明确产物、来源引用和审核后的共享知识完成。

好的记忆系统，不是让 Agent 什么都记住。

而是让它知道：

- 什么值得记。
- 应该记到哪里。
- 什么时候不能记。
- 谁有权读取。
- 谁有权写入。

## 四条边界：上下文、记忆、权限、验收

多 Agent 系统真正难的地方，往往不是 Agent 不够聪明，而是边界太软。

![Agent Boundaries](https://chilohdata.s3.bitiful.net/blog/blog/agent-boundaries.svg)

### 1. 上下文边界

不是所有信息都应该常驻上下文。

常驻内容越多，噪声越大，真正关键的信息越容易被淹没。

我更倾向于：

```text
常驻：核心规则、角色边界、禁止项
按需：方法、模板、案例、历史资料
隔离：大规模检索、并行探索、长文档阅读
不进：确定性校验、审计日志、可重新读取的大输出
```

上下文不是仓库。

上下文是当前任务的工作内存。

### 2. 记忆边界

项目事实归项目。

角色经验归角色。

共享方法进共享知识库。

外部资料作为资料，不作为指令。

这几条如果混了，系统很快就会开始“带着旧伤理解新问题”。

### 3. 权限边界

能读，不代表能写。

能生成，不代表能发布。

能构造 payload，不代表能调用外部 API。

涉及删除、覆盖、发送、支付、部署这类动作，必须有明确审批。

越自动化，越要知道什么时候停。

### 4. 验收边界

完成不能靠“看起来差不多”。

一个结果至少要说清楚：

- 目标是否满足。
- 边界是否遵守。
- 风险是否披露。
- 证据是否足够。
- 下游是否能接着用。

如果缺用户决策，就不是完成。

如果证据不足，就不是完成。

如果风险没有披露，也不是完成。

表达可以很漂亮，但验收必须很冷静。

## 事件流：让协作可以恢复

多人协作需要事件流，多 Agent 也一样。

事件流不一定一开始就很复杂，但至少要记录这些事情：

| 事件 | 为什么重要 |
|---|---|
| 用户目标更新 | 防止还按旧目标执行 |
| Worker 被选中 | 知道谁负责 |
| 任务被认领 | 避免重复劳动 |
| 子任务被创建 | 控制并行和预算 |
| 阻塞被提出 | 及时停下来 |
| 风险被记录 | 最终交付时不遗漏 |
| 产物准备好 | 下游可以接手 |
| 验收通过或失败 | 完成有依据 |
| 记忆写入建议 | 防止随意污染长期记忆 |

没有事件流，系统只能靠聊天记录回忆。

聊天记录太长、太杂、太容易被压缩，也太容易被重新解释。

事件流的价值，是让任务中断后能恢复，出错后能复盘，冲突时能回到事实。

## 一个可用的运行过程

把上面的对象串起来，一次任务大概会这样跑：

```text
User 提出目标
  -> Channel 记录任务和约束
  -> 生成本次任务需要的上下文包
  -> Main Agent 做路由和风险判断
  -> Worker Agent 接任务并执行
  -> 必要时调用 Task Subagent 做局部并行
  -> Worker 验证并整合结果
  -> Channel 收到结构化产物
  -> Main 只在需要时做仲裁、验收和最终整合
  -> 合格内容再进入项目记忆或共享知识
```

这里有两个重点。

第一，共享上下文尽量只召回一次。

不要 Main 召回一遍，Worker 召回一遍，Subagent 又召回一遍。这样不仅浪费，也容易让每个角色拿到不同版本的事实。

第二，最终产物优先回到 Channel。

不要让每个 Worker 的结果都靠 Main 反复改写。能结构化交付，就结构化交付；需要整合时，Main 再做整合。

## 常见反模式

我现在最警惕这几种设计：

| 反模式 | 后果 |
|---|---|
| Main Agent 什么都亲自做 | 多 Agent 退化成单 Agent |
| Worker 没有拒绝边界 | 什么任务都接，责任不可定位 |
| Subagent 可以继续开 Subagent | 治理失控，目标漂移 |
| 长期记忆随手写 | 项目事实、偏好、猜测混在一起 |
| 工具权限只看能力不看风险 | 外部副作用不可追踪 |
| 没有验收状态 | “写完了”被误当成“交付了” |
| 所有信息都进上下文 | 噪声越来越大，压缩后丢关键决策 |

这些问题看起来都是工程细节，但最后都会变成体验问题。

用户感受到的不是“架构边界不清”，而是：

- 它怎么又忘了？
- 它怎么擅自改了？
- 它怎么前后说法不一样？
- 它怎么做完了但不能用？

## 最后

我越来越觉得，Agent 的下一阶段不是单体能力继续堆高，而是协作能力变成熟。

单个 Agent 当然会越来越强。

但只要进入真实项目，就会遇到任务流转、角色分工、权限约束、状态恢复、风险披露和质量验收。

这些不是模型参数自己能解决的。

它们需要组织设计。

如果说早期 AI 产品解决的是“模型能不能回答”，`MCP` 解决的是“模型能不能接上现实”，`Skills` 解决的是“模型知不知道怎么做”，`CLI` 解决的是“模型能不能执行和验证”，那多 Agent 要解决的就是：

**多个智能体能不能像一个可信团队一样协作。**

不是每个 Agent 都要无所不能。

相反，越成熟的系统，越应该让每个 Agent 知道：

- 自己负责什么。
- 不负责什么。
- 什么时候出手。
- 什么时候交还。
- 什么可以沉淀。
- 什么必须停下。

多 Agent 真正的分水岭，不是数量。

而是组织。
