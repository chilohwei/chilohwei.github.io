## 太长不读

最近做一条偏交付向的内容 Workflow，踩完一圈后，我对 `Skill Workflow` 的理解变得很明确：

**不要把它当成更长的提示词，要把它当成一套小型工作系统。**

一个能稳定交付的 Workflow，至少要回答五件事：

| 问题 | 靠什么解决 |
|---|---|
| 什么时候该启动 | 清晰的触发条件 |
| 现在该做哪一步 | 状态机和唯一下一步 |
| 哪些判断可信 | 证据账本和来源引用 |
| 产物能不能被用户判断 | 可读预览 |
| 哪些动作不能自动做 | 审批门禁和安全交付 |

模型很适合做理解、取舍、表达，但不适合承担所有确定性工作。凡是涉及校验、状态、格式转换、发布前检查、外部写入，最好交给脚本和门禁。

![Skill Workflow Loop](https://chilohdata.s3.bitiful.net/blog/blog/skill-workflow-loop.svg)

这篇不是讲“怎么写一个很厉害的 prompt”，而是讲我现在会怎样设计一个能交付的 Skill Workflow。

## 先把概念说清楚

我以前也容易把 `Skill` 理解成提示词增强包。

比如在 `SKILL.md` 里写清楚角色、语气、步骤、注意事项、输出格式，然后期待 Agent 每次照着做。

这在单点任务里没问题，比如总结一篇文章、改写一段文案、生成一个模板。

但一旦任务变成多阶段交付，它就不够了。

比如一个内容 Workflow，看起来只是“让 AI 写文章”，实际链路可能是：

```text
判断主题是否值得写
  -> 收集资料和证据
  -> 生成内容策略
  -> 写作和修改
  -> 生成预览
  -> 等用户审批
  -> 构造平台草稿
  -> 发布前校验
```

写作只是中间一环。

真正麻烦的是：系统怎么知道自己在哪一步，下一步该做什么，哪些数据不够，哪些动作必须停下来问用户。

所以我现在会这样区分：

| 类型 | 本质 | 适合做什么 | 容易犯的错 |
|---|---|---|---|
| Prompt | 一次表达约束 | 单次生成、改写、总结 | 越写越长，最后没人看得清 |
| Skill | 按需加载的方法包 | 给 Agent 一套做事方法 | 把百科和脚本全塞进正文 |
| Workflow | 可推进的业务流程 | 多阶段、可恢复、可交付任务 | 没状态、没证据、没审批 |

一句话：**Prompt 解决一次输出，Workflow 解决一段职责。**

## 一个 Workflow 至少分五层

我现在不太会一上来写 `SKILL.md`。

我会先拆层。

![Skill Workflow Layers](https://chilohdata.s3.bitiful.net/blog/blog/skill-workflow-layers.svg)

| 层 | 负责什么 | 不应该负责什么 |
|---|---|---|
| 入口层 | 识别任务、明确输入、给出第一步 | 承载所有领域知识 |
| 知识层 | 方法论、模板、案例、判断框架 | 常驻进上下文污染每次任务 |
| 执行层 | 脚本、校验、格式转换、预览生成 | 让模型凭感觉判断确定性结果 |
| 状态层 | 进度、证据、决策、变更记录 | 依赖聊天记录记住一切 |
| 交付层 | 预览、审批、发布、回滚 | 把外部副作用藏进“顺手做了” |

这五层不一定每个项目都要做满。

内部草稿、一次性小工具，可能一个 `SKILL.md` 就够。

但只要涉及客户交付、外部平台、多人协作、长期状态，就不能只靠聊天上下文。

## 模型和脚本的分工

这件事里最重要的判断，是哪些东西该让模型做，哪些东西不该让模型做。

我现在的经验大概是这样：

| 任务 | 更适合模型 | 更适合脚本 |
|---|---|---|
| 理解用户目标 | 是 | 否 |
| 判断优先级 | 是 | 否 |
| 生成结构和表达 | 是 | 否 |
| 校验文件是否存在 | 否 | 是 |
| 检查字段是否齐全 | 否 | 是 |
| 生成预览 | 部分 | 是 |
| 判断证据是否足够 | 部分 | 部分 |
| 发布、覆盖、写入外部系统 | 否 | 是，但必须审批 |

模型的问题不是“不聪明”。

模型的问题是它太会解释了。

缺字段，它能解释成“可能暂时不需要”；证据不足，它能写成“基于目前信息可以推测”；预览没生成，它也能说“内容结构已经准备好”。

但交付系统不能靠这种弹性。

一个脚本返回：

```json
{
  "status": "blocked",
  "stage": "data_readiness",
  "summary": "缺少正式生产所需的资料来源。",
  "next_action": "补齐品牌资料或允许降级为草稿。",
  "safe_to_continue": false
}
```

这比一段好看的解释有用多了。

因为它告诉 Agent：现在不能装作完成。

## 状态机不是复杂化，是防止失忆

复杂 Workflow 最大的问题，是它会跨很多轮对话。

用户今天补一点资料，明天改一下方向，后天又让你接着做。如果所有状态都靠聊天记录，Agent 迟早会猜错。

所以状态要落盘。

不用一开始就搞很重，一个轻量目录已经能解决很多问题：

```text
outputs/[task-id]/
  workflow-state.yaml
  evidence-ledger.yaml
  decision-log.yaml
  change-log.yaml
  preview/
  artifacts/
```

各文件职责要分清：

| 文件 | 保存什么 | 为什么需要 |
|---|---|---|
| `workflow-state.yaml` | 当前阶段、阻塞、下一步 | 中断后能恢复 |
| `evidence-ledger.yaml` | 来源、时间、置信度 | 关键结论能追溯 |
| `decision-log.yaml` | 用户确认过什么 | 避免反复问和擅自改 |
| `change-log.yaml` | 改过什么、为什么改 | 方便复盘和回滚 |
| `preview/` | 用户能看的版本 | 审批基于可见产物 |

这不是为了显得工程化。

这是为了让 Agent 不用每次从长对话里重新猜一遍。

## 三个硬门禁：证据、预览、审批

我现在判断一个 Workflow 是否靠谱，会先看这三道门。

### 1. 证据门禁

关键结论必须能回到来源。

不同业务的证据不一样：

| 场景 | 证据可能是什么 |
|---|---|
| 内容策略 | 搜索量、竞品页面、用户评论、品牌资料 |
| 财务分析 | 原始账表、科目映射、汇率来源 |
| 客服诊断 | 工单记录、产品版本、复现步骤 |
| 产品需求 | 用户反馈、业务目标、约束条件 |

证据不足，就降级成草案、方向建议或待验证假设。

不要把“估计”包装成“验证通过”。

### 2. 预览门禁

用户审批的应该是可读预览，不是 Markdown、JSON 或聊天摘要。

尤其是内容、报告、网页、CMS 草稿，预览不是装饰，而是决策界面。

没有预览，用户很难真正判断产物。

### 3. 审批门禁

发布、覆盖、删除、排期、写入外部系统，这些动作必须单独确认。

生成 payload 不等于可以写入平台。

打开预览不等于用户批准发布。

这条边界一定要硬。

越强的 Agent，越需要明确的刹车。

## 用户体验，其实是流程控制协议

以前我会把用户体验理解成“回复写得友好一点”。

现在我更愿意把它看成 Workflow 的控制协议。

用户不关心内部有多少脚本、多少状态文件、多少校验步骤。

用户真正关心四件事：

| 用户问题 | 好回复应该做到 |
|---|---|
| 现在到哪了 | 给当前阶段，不倒内部流水账 |
| 为什么卡住 | 先讲影响，再讲原因 |
| 结果可信吗 | 给证据状态和风险 |
| 我下一步做什么 | 只给一个明确动作 |

一个好的 Workflow 回复，应该遵守几个小原则：

- 直接问题直接答，不借机展开整套系统。
- 一轮回复只服务一个场景，不把安装、诊断、审批、发布混在一起。
- 失败时先说影响，再说恢复动作。
- 需要用户判断时停下来，不要替用户决定。
- 下一步尽量唯一，别把用户淹没在选项里。

这不是文案技巧。

这是权限边界。

如果用户体验不被工程化，Agent 很容易在“我很积极”的状态里越权。

## 我会复用的五个模式

这几个模式换行业也成立。

| 模式 | 解决什么 |
|---|---|
| 唯一下一步裁判 | 根据状态告诉 Agent 当前唯一该做什么 |
| 证据账本 | 让结论能回到来源、时间和置信度 |
| 预览即门禁 | 用户批准的是可见产物，不是聊天总结 |
| 配置自愈 | 缺配置就生成模板，缺字段就提示本轮必填项 |
| 安全发布适配器 | 构造 payload → 校验 → 预览 → 确认 → 写入 |

但这些是模式库，不是全家桶。

开得越多，reference 越长，路径越碎，默认就更慢、更费 token。

我的建议是：**缺什么风险，补什么模块。**

不是每个 Workflow 都需要完整证据系统、完整 doctor、完整回滚链路。

但涉及不可逆动作、外部系统、客户交付时，该有的门禁不能省。

## 从零设计，我会先问九个问题

如果今天重新设计一个 Skill Workflow，我不会先写文档。

我会先问：

1. 这个 Workflow 最终交付什么？
2. 谁会使用它？
3. 用户最容易卡在哪里？
4. 哪些步骤允许模型判断？
5. 哪些步骤必须脚本化？
6. 哪些中间产物必须落盘？
7. 哪些数据是正式生产门槛？
8. 哪些动作有外部副作用，必须审批？
9. 中断、失败、修改之后，系统怎么恢复？

这些问题答完，目录结构自然会长出来。

一个相对完整的版本可能是：

```text
your-skill-workflow/
  SKILL.md
  references/
    operating-principles.md
    workflow-stages.md
    preview-and-approval.md
  scripts/
    workflow_start.mjs
    workflow_next.mjs
    validate_evidence.mjs
    render_preview.mjs
    delivery_validate.mjs
  assets/
    templates/
```

重点不是目录漂亮。

重点是每一层职责清晰：

- `SKILL.md` 做入口和导航。
- `references/` 放详细规则和案例。
- `scripts/` 做确定性动作。
- `assets/` 放模板和可复用材料。
- 状态文件记录进度、证据、决策和变更。

不要把所有东西塞进一个巨型 `SKILL.md`。

那不是沉淀，是上下文污染。

## 最后

`MCP` 让 Agent 接上现实。

`Skills` 让 Agent 学会一套做事方法。

`CLI` 让 Agent 能执行、验证、修改、再执行。

而 `Skill Workflow` 要做的，是把这些东西组织成一段可交付的业务流程。

它不是让 AI 更会说。

它是让 AI 更知道：

- 现在处在哪个阶段。
- 哪些信息可信。
- 哪些动作可以继续。
- 哪些动作必须停止。
- 结果如何被用户看见。
- 外部副作用如何被确认。

提示词解决一次输出。

Workflow 解决一段职责。

当 AI 开始承担职责，我们真正要设计的，就不再只是“怎么让它答得更好”。

而是：

**怎么让它在真实世界里，把事做对。**
