> 这篇文章只讲一个判断：**高质量 Skill Workflow 不是一份更长的 prompt，而是一套工作系统。** 模型负责理解与创作；脚本负责校验、状态与交付前检查；状态机负责可恢复、可审计。
>
> 结果好不好，往往不取决于「AI 会不会写」，而取决于「系统能不能交付」。**「工作系统」也不等于「样样检查做满」**——门禁、reference 的厚度要和真实风险、真实节奏匹配，否则瓶颈的就是你自己。

最近在把一个面向「内容策略—写作—预览—多平台发布」的长链路 Workflow 做实的过程中，我对 `Skill Workflow` 这件事的理解变得更清楚了一些。

一开始很容易把它理解成“把提示词写得更细一点”。

比如把角色、语气、步骤、注意事项、输出格式全都塞进一个 `SKILL.md` 里，然后期待 Agent 每次都能照着做。

但真正做过一轮复杂流程之后，会发现这条路很快就会走到尽头。

因为真实业务里，问题从来不只是“生成一段内容”。

真正的问题是：

- 用户的目标是否清楚？
- 数据是否足够可信？
- 哪些地方可以让模型判断？
- 哪些地方必须由脚本验证？
- 产物能不能被预览？
- 发布、覆盖、写入这些动作有没有审批？
- 中断之后能不能恢复？
- 出错之后能不能追溯？

这些问题，靠加长 prompt 解决不了，只能靠系统设计。

## 一、从写内容，到推动流程

以这类「内容 + 多平台交付」的 Workflow 为例，它表面上看是在做内容生产。

但它真正要解决的，并不是“让 AI 写一篇文章”。

如果只是写文章，任何一个大模型都能写。

真正难的是：

```text
判断主题是否值得写
  -> 收集真实数据
  -> 生成内容策略
  -> 写作和优化
  -> 准备图片与媒体清单
  -> 生成 HTML 预览
  -> 等待用户审批
  -> 构建 WordPress / Shopify 草稿包
  -> 发布前校验
```

这一整条链路里，写作只是一环。

甚至很多时候，写作不是最难的部分。

更难的是：**这个系统怎么知道自己现在应该做什么，不应该做什么，以及什么时候必须停下来等用户确认。**

这也是我这次最深的感受：

**不要把 Agent 当成“会写文章的模型”，要把它设计成“能推动业务流程的操作系统”。**

一旦视角切换过来，设计重点就变了。

你不再只问“提示词怎么写得更好”，而是开始问：

- 这个 Workflow 最终交付什么？
- 用户是谁？
- 哪些步骤必须有真实数据？
- 哪些动作会产生外部副作用？
- 哪些结果必须可预览？
- 哪些判断必须被记录？
- 失败时用户应该看到什么？

这才是 Workflow 的起点。

## 二、高标准 Workflow 的七个条件

一个普通 Skill，往往解决一个单点任务。

比如处理 PDF、生成摘要、调用某个 API、转换某种格式。

但一个高标准的 `Skill Workflow`，解决的是跨阶段、跨工具、跨决策的业务流程。

我现在会用七个条件判断它是否成立：

**第一，可触发。**

用户用自然语言描述目标时，系统能准确知道应该调用它，而不是靠用户记住复杂命令。

**第二，可执行。**

用户不需要理解内部技术细节，Agent 能自动推进大部分步骤。

**第三，可验证。**

关键阶段不能只靠“看起来不错”。它应该有脚本、文件、状态、测试或预览证明结果可信。

**第四，可恢复。**

中断后能从状态文件继续，而不是靠聊天记录里的记忆继续。

**第五，可预览。**

凡是会影响用户判断的产物，都应该能被看见。

**第六，可审计。**

数据来源、用户审批、自动修改、发布动作，都应该有记录。

**第七，可交付。**

它不能停在“我给你一段建议”。它要能走到草稿包、平台草稿、客户交付物，或者至少是一个可检查的本地产物。

这七个条件叠在一起，`Skill Workflow` 就更接近轻量产品系统，而不是堆提示词。

但它们是一张**上限清单**，不必第一版全盘兑现。若是周报短文、内部草稿、人工扫一眼即可的产出，硬上「全链路证据 + 多层 doctor + 全套 reference」，往往得不偿失：阻断步骤一多，主线就慢；Agent 反复加载长文档、串联检查，上下文膨胀，**大量 token 花在维持流程上，并不换成交付质量**。

设计前先定优先级：最怕**不可靠**，还是**太慢 / 太贵（含模型成本）**？先锁住最怕的一项，其余按事故与反馈迭代，别一上来按「教科书目录」搭满。

## 三、模型做判断，脚本做确定性

这件事里最重要的分工，是把模型和脚本的职责分清楚。

模型很适合做这些事：

- 理解用户的自然语言目标。
- 判断业务语境和优先级。
- 生成内容结构、文案和解释。
- 在多个合理方案之间做取舍。
- 用人能理解的方式说明当前状态。

但模型不适合承担所有事情。

凡是需要稳定、重复、可测试、不能靠感觉的地方，都应该尽量脚本化。

比如：

- 初始化工作区。
- 判断下一步。
- 检查配置是否存在。
- 校验证据是否足够。
- 生成预览。
- 构建平台 payload。
- 做发布前检查。
- 检查回复是否暴露了过多内部细节。

一个好的脚本，应该像一个小型 API。

它不需要交互式输入，有清楚的参数，有 `--help`，能输出稳定的 JSON 或稳定文本，失败时能说明 blocker 和 recovery。

比如它可以告诉 Agent：

```json
{
  "status": "blocked",
  "stage": "data_readiness",
  "summary": "外部数据源凭证未配置或不可用。",
  "one_next_action": "在本地工作流的密钥文件中补齐对应 API 凭证（勿粘贴到对话里）。",
  "safe_to_continue": false
}
```

这看起来只是工程细节，但它会直接影响用户体验。

因为 Agent 的回复，很多时候不是凭空写出来的，而是根据这些稳定状态组织出来的。

**脚本越稳定，Agent 越不容易胡来。**

分寸在于：**多一道校验就多一轮读写与等待**；把脚本压在「不可逆动作」和「真翻过车的环节」上，比每次全套体检更符合协作体感——这点与上一节的「上限清单」是一条线上的两件事。

## 四、状态机，是复杂流程的记忆

复杂 Workflow 不能只靠对话记忆。

阶段一多、来回一改再改，就更值得用落盘状态接住；涉及审批、对外写入或多人协作时，日志与决策记录也更值得前置——都不是为了「显得专业」，而是为了中断可续、责任可追溯。

在这类项目里，一个业务对象（例如品牌/客户）的工作区大概会长这样：

```text
outputs/[业务标识]/
  profile.yaml
  workflow-state.yaml
  evidence-ledger.yaml
  decision-log.yaml
  change-log.yaml
  credentials-status.json
  data/
  content-assets/
```

它们解决的是很现实的问题：

- 用户中断后，系统知道从哪里继续。
- Agent 不会跳过前置门禁。
- 用户审批过什么，拒绝过什么，有记录。
- 某个结论来自哪份数据，可以追溯。
- 多个品牌、多个项目之间，不会互相污染。

很多时候，Agent 出错不是因为它“不聪明”，而是因为流程没有给它一个可靠的状态来源。

如果它只能从上下文里猜，那它迟早会猜错。

状态文件的价值，就是让 Workflow 从“对话里的临时过程”变成“磁盘上的持续过程”。

## 五、证据、预览、审批，是三道硬门禁

做内容、报告、数据分析、投放策略、法务文档，其实都有同一个问题：

**不能把“估计”伪装成“验证通过”。**

这里的语境是：**一旦有对外交付、合规或追责后果，证据就不能含糊**；随手纪要、内部试水仍可沿用第二节说的按需降级。在此基础上，高质量 Workflow 才谈得上证据门禁。

没有证据，就不要做最终判断。

证据不足，就降级成诊断、方向建议或草案。

证据完整，再进入正式生产。

对内容生产来说，证据可能是搜索量、竞品、AI 可见度、品牌资料。

对财务分析来说，证据可能是原始账表、科目映射、时间范围和汇率来源。

对客户支持来说，证据可能是工单记录、产品版本、复现步骤和用户环境。

不同业务的证据不一样，但原则一样：

**关键结论必须能回到来源。**

除了证据门禁，还有预览门禁。

Markdown、YAML、JSON 都可以是源文件，但用户真正审批的，应该是可读的预览。

尤其是内容、报告、网页、CMS 草稿这类产物，预览不是装饰，而是决策界面。

最后是审批门禁。

发布、覆盖、删除、排期、写入外部系统，这些动作必须单独确认。

打开预览，不等于用户批准发布。

生成 payload，不等于可以写入平台。

这条边界必须清楚。

因为越是强的 Agent，越需要明确的刹车。

## 六、用户体验不是润色，是流程控制协议

以前我会把用户体验理解成文案是否友好、表达是否清楚。

现在我更倾向于把它看成 Workflow 的控制协议。

用户不关心 `workflow_start.mjs`、`workflow_next.mjs`、`evidence-ledger.yaml` 这些内部结构。

用户真正关心的是：

- 现在卡在哪里？
- Agent 做了什么？
- 结果可信吗？
- 我下一步只需要做什么？

所以一个好的 Workflow 回复，不应该把所有内部复杂度都倒给用户。

它应该遵守几个很朴素的原则：

**直接问题，直接回答。**

用户问“配置文件在哪”，就告诉他路径，不要展开八步流程。

**一轮回复，只服务一个场景。**

安装、诊断、生产、审批、发布，不要混在一起。

**一轮回复，只给一个下一步。**

用户最需要的往往不是更多选项，而是明确知道自己现在该做什么。

**失败时先讲影响，再讲恢复动作。**

不要先贴一大段堆栈，不要先把责任丢给用户。

**审批门禁必须停下来。**

到了需要用户判断的地方，就不要继续替用户做决定。

这不是文案技巧，而是系统边界。

用户体验如果不被工程化，Agent 很容易在“看起来很积极”的状态里越权。

## 七、五个值得复用的模式

抽象一层之后，这几件事换行业也成立。

**唯一下一步裁判（Arbiter）。** 用脚本根据状态给出「当前唯一该做的动作」，避免每轮自由发挥把用户淹没在选项里。

**证据账本。** 关键结论要回到来源、时间、原始材料与置信度；写策略、做研究、出报告都同理。

**预览即门禁。** Markdown、YAML 可以是源文件，但审批界面应是可读预览；用户批准的是预览，不是聊天记录摘要。

**配置自愈。** 缺配置文件就生成模板，缺字段就只提示「这一轮目标需要填哪一组」，密钥永远落在本地，不进对话。

**安全发布适配器。** 外部写入一律拆成：构造 payload → 校验 → 预览 → dry-run 或草稿 → 明示确认 → 再写入。对 CMS、邮件、数据库都适用。

这是**模式库**，不是任务清单。**开得越多，reference 越长、路径越碎，默认就更慢、更费 token**——按场景裁剪，比套用全套重要。

## 八、如果从零做一个 Workflow

如果今天重新从零设计一个 `Skill Workflow`，我大概不会先写 `SKILL.md`。

我会先回答九个问题：

1. 这个 Workflow 最终要交付什么？
2. 谁会使用它？
3. 用户最容易卡住的地方是什么？
4. 哪些步骤允许模型判断？
5. 哪些步骤必须用脚本确定性处理？
6. 哪些中间产物必须落盘？
7. 哪些数据是正式生产门槛？
8. 哪些动作有外部副作用，必须审批？
9. 中断、失败、修改之后，系统如何恢复？

这九个问题答完，才开始拆结构。

一个相对成熟的目录，大概会是这样：

```text
your-skill-workflow/
  SKILL.md
  references/
    operating-principles.md
    user-experience.md
    configuration.md
    data-sources.md
    workflow-stages.md
    preview-and-approval.md
    publishing-or-delivery.md
  scripts/
    workflow_start.mjs
    workflow_next.mjs
    state_init.mjs
    data_readiness.mjs
    validate_evidence.mjs
    render_preview.mjs
    artifact_manifest.mjs
    delivery_validate.mjs
    workflow_doctor.mjs
    response_lint.mjs
  assets/
    templates/
```

不是每个 Workflow 都需要这么完整。

单点任务可能只有一个 `SKILL.md` 就够。

多阶段任务至少要有 `references/operating-principles.md` 和状态脚本。

涉及外部平台，就必须有 dry-run、payload validate 和 approval gate。

面向非技术用户，就要有配置引导和运行时自检。

面向交付或客户，通常还需要 doctor、测试与复盘闭环——但 **doctor 仍要克制**（与第二节「上限清单」、第七节「模式库」同一条逻辑）：适合盯住包结构、致命缺配、以及写入外部系统前的硬闸门；验证若拆得过碎，用户永远在等对勾，Agent 则反复吞 `references` 和诊断输出，token 耗在流程本身。其余用轻量校验，或合并进少数几步里做完。

核心不是目录漂亮，而是每一层职责清晰、厚度与风险匹配。

`SKILL.md` 像入口和路标。

`references/` 放详细规则。

`scripts/` 放稳定动作。

`assets/` 放模板和可复用材料。

不要把所有东西都塞进一个巨型 `SKILL.md`——那是上下文污染，不是沉淀。

常被忽视的反面同样成立：只靠对话记流程、缺数据仍产出所谓 final、把预览当审批——等于把确定性交给运气。该落脚本、状态和门禁的，就别停在文案层。

## 九、从 Prompt 到 Workflow，是 AI 产品化的分水岭

如果把这件事放到更大的 AI 演化里看，它其实和我之前对 `MCP -> Skills -> CLI` 的理解是一脉相承的。

`MCP` 解决接入。

`Skills` 解决胜任。

`CLI` 解决闭环。

而 `Skill Workflow` 做的，是把这三件事组织成一个可交付的业务过程。

它不是让 AI 变得更会说。

它是让 AI 更知道：

- 现在处在哪个阶段。
- 该调用什么能力。
- 什么时候应该继续。
- 什么时候必须停止。
- 什么结果可以相信。
- 什么动作需要审批。

从这个角度看，未来很多 AI 产品的壁垒，可能不只在模型本身。

模型当然重要。

但真正拉开差距的，可能是围绕模型建立起来的工作系统：

```text
清晰触发入口
  -> 渐进加载的知识结构
  -> 显式状态机
  -> 确定性脚本
  -> 可审计证据
  -> 可预览产物
  -> 用户审批门禁
  -> 安全交付与复盘
```

这是一条「完整链路」示意，不是一次性盖完的工期表：**缺什么风险就补什么模块**，比一口吃成胖子更符合真实迭代。

这套东西看起来没有模型能力那么性感。

但它决定了 AI 能不能从“回答问题”走向“承担工作”。

提示词解决的是一次输出。

Workflow 解决的是一段职责。

而当 AI 开始承担职责，我们真正要设计的，就不再只是“怎么让它答得更好”。

而是：

**怎么让它在真实世界里，把事做对。**
